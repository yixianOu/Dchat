# 论文简化版 —— 逻辑与实现要点梳理

> 以下按论文原有逻辑主线，压缩为功能和技术实现的核心论证链，约 250 行。

## 一、问题定义

主流 IM（微信、WhatsApp）均为中心化架构，三个根本问题：
1. 数据主权旁落 —— 聊天记录、联系人存储在服务端，用户无控制权。
2. 元数据泄露 —— 服务端可获取通信双方身份、通信时间。
3. 单点故障 —— 服务器宕机则全网不可用。

现有去中心化方案各有短板：
- 纯 P2P（Tox）：NAT 穿透困难、离线消息无法投递。
- 联邦式（Matrix）：数据仍存服务器，未根本解决数据主权。

→ 论文提出的核心命题：**能否既有 P2P 的数据主权，又有 C/S 架构的连接稳定性？**

## 二、方案设计

**核心思路："C/S 连接，本地存储"。** 网络连接用 C/S 模式（保证稳定），数据全放本地（保证主权）。服务端只转发加密消息，不存任何业务数据。

技术选型：
- **NATS** 作为消息中间件：发布/订阅模式、LeafNode 层级扩展、Routes 集群、JetStream 离线持久化。
- **NaCl Box**（X25519 + XSalsa20-Poly1305）做私聊端到端加密。
- **AES-256-GCM** 做群聊对称加密。
- **SQLite** 做本地存储。
- **Wails**（Go + React）做跨平台桌面应用。

## 三、架构设计

```
┌─ 应用层（React UI + Go 业务逻辑）
│  └─ 加密、SQLite、NATS 客户端
├─ NATS 客户端层（连本地 LeafNode 127.0.0.1:4222）
├─ LeafNode 层（应用内置 NATS Server，仅本机回环）
├─ 集群层（公网 Hub 全 mesh Routes，消息广播）
└─ 认证层（NSC JWT：Operator/Account 固定种子 + User 独立密钥）
```

**为什么加 LeafNode？** 用户设备不需要公网 IP 和入站端口，只需出站连接 Hub。LeafNode 对本地客户端透明，NATS 标准客户端直接可用。

**Hub 为什么用 Routes 全 mesh？** 每个 Hub 与其他所有 Hub 直连，消息发布到任一节点自动广播全集群。节点地位平等，无单点。缺点是连接数为 O(n²)，且 JetStream 下集群节点需固定配置（Raft 共识限制）。

## 四、关键实现细节

### 4.1 消息主题

- 私聊：`dchat.dm.<cid>.msg`，cid = SHA-256(排序后的两个用户 ID)[:16]
- 群聊：`dchat.grp.<gid>.msg`，gid 为随机生成

### 4.2 双密钥体系（Ed25519 + X25519）

| 层次 | 算法 | 用途 |
|------|------|------|
| 身份认证 | Ed25519 | NATS JWT 签名验证 |
| 消息加密 | X25519 | ECDH 密钥交换 |

X25519 私钥从 Ed25519 私钥确定性单向派生（基于曲线同源特性），**不可逆**。用户只需一个种子即可同时完成认证和加密，但两种密钥密码学上相互独立——签名密钥泄露不影响历史加密消息。

### 4.3 私聊加密（NaCl Box）

```
发送：临时X25519密钥对 → ECDH共享密钥 → XSalsa20-Poly1305加密 → 组装EncWire JSON
字段：{cid, sender, ts, nonce(Base64), cipher(Base64), nickname}
接收：提取sender → 查本地好友公钥 → ECDH共享密钥 → 解密 → 验证MAC → 存入SQLite
```

安全特性：临时密钥提供**前向安全性**；Poly1305 MAC 保证**完整性**；常量时间算法防**时序侧信道**。

### 4.4 群聊加密（AES-256-GCM）

群聊无法一对一 ECDH（开销太大），改用**共享对称密钥**。群主用 crypto/rand 生成 32 字节密钥，通过 NaCl Box 逐条私聊分发给成员。加密用 AES-256-GCM + 随机 Nonce。

**安全妥协**：成员设备被攻破则全群历史可解密；退出成员仍持密钥（当前未做密钥轮换）。

### 4.5 本地存储

SQLite 四表：conversations（会话）、messages（消息明文）、friend_pub_keys（好友公钥）、group_sym_keys（群密钥）。

私钥文件全部 600 权限，仅当前用户可读，存于 ~/.dchat/。

### 4.6 离线消息

JetStream 按用户创建独立 Stream/Consumer，离线消息暂存 Hub。上线后 Pull 批量拉取（每批 100 条），逐条解密存入 SQLite 后发送 ACK，Hub 标记已消费后移除。

去重：靠消息 ID 查询本地数据库，已存在则跳过。

### 4.7 应用启动（链式依赖）

```
加载配置 → 启动 LeafNode → 初始化 SQLite → NATS 客户端连接 → 创建聊天服务 → 恢复历史会话
```

配置缺失或数据库损坏会导致中断；LeafNode 端口冲突或 Hub 不可达自动切换并记录日志。

### 4.8 前端与通信

React + Tailwind，左右分栏（会话列表 + 聊天区）。前后端通过 Wails 绑定通信：前端调 Go 方法发消息，后端通过 EventsEmit 推送事件更新 UI。

## 五、测试验证

### 功能测试（15 项全部通过）
涵盖：密钥生成/导入导出、LeafNode 启动与连接、JWT 认证、私聊/群聊收发、跨 Hub 传输、离线存储/同步/去重、SQLite 读写、历史分页与搜索。

### 安全验证
1. **端到端加密**：nats sub 监听到的 cipher 字段为 Base64 密文，缺私钥无法解密。
2. **密钥本地存储**：~/.dchat/ 下文件权限 600，服务端无任何私钥。
3. **服务端无数据**：Hub 上 find 搜索无 user/chat/message 相关明文文件。
