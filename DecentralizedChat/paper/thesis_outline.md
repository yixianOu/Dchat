#### 4.1.3 Hub认证机制（400字）

**内容要点**：

- **默认模式（无鉴权）**：
  - 便于测试和公网共享部署
  - 任何LeafNode均可连接，降低使用门槛
  
- **JWT认证模式（可选，私有部署）**：
  - 基于NSC（NATS Security Center）的JWT认证体系
  - Hub配置`simple_resolver.conf`加载账户信息
  - LeafNode连接时携带`user.creds`凭证
  - 服务端验证JWT后才允许连接
  
- **部署方式**：将本地`~/.dchat/`的resolver配置和accounts目录复制到服务器

**安全建议**：

- Operator私钥（`operator.nk`）必须离线保存
- 仅复制公开配置到服务器，私钥留在本地

---

#### 4.1.4 Hub集群Routes配置（500字）

**内容要点**：

- 公网Hub通过Routes形成固定集群（当前实现）
- 全mesh网络：集群内每个Hub与其他所有Hub直连
- 消息广播：发布到任一Hub，自动广播到集群内所有Hub
- **集群边界**：新的NATS节点无法动态加入（JetStream Raft算法的限制）

**技术取舍**：

- 若**不支持离线消息**（不用JetStream）：纯NATS的Routes集群支持动态新增节点，新节点可链式连接到一个节点后自动发现集群内其他节点
- 若**支持离线消息**（使用JetStream）：集群节点必须固定，新节点无法动态加入（Raft共识限制）
- **核心限制**：无论哪种方式，都无法突破NAT穿透实现真正的P2P通信

**去中心化体现**：

- 集群内多个Hub地位平等，无主节点
- 任一Hub故障不影响集群整体服务
- 集群可水平扩展（需预先配置）

**未来扩展**：通过Gateway连接不同的公网Routes集群，实现跨集群消息转发

### 4.2 加密与安全模块（1,200字）

#### 4.2.1 密钥体系（400字）

**内容要点**：

- Ed25519签名密钥对（身份标识）
- X25519加密密钥对（ECDH密钥交换）
- 本地存储：`user.seed`、`user.creds`

**密钥派生原理**：

本项目采用 Ed25519 作为身份签名密钥（用于 NATS JWT 认证），但**不直接使用其进行消息加密**。原因如下：首先，Ed25519 是专为数字签名设计的曲线，不具备直接加密能力；其次，根据密码学最佳实践，签名密钥与加密密钥应分离，避免单点失效导致既丢失身份认证又泄露通信内容；最后，通过 Ed25519 私钥派生 X25519 密钥对（基于曲线同源特性），既保持身份一致性，又获得 ECDH 密钥交换能力，实现端到端加密。这种设计确保即使签名密钥意外泄露，历史会话的加密消息仍无法被解密。

**对应代码**：`internal/chat/nsc_crypto.go`

#### 4.2.2 私聊加密实现（500字）

**内容要点**：

1. 生成临时X25519密钥对
2. ECDH计算共享密钥
3. NaCl Box加密
4. 构造消息载荷（EncWire）

**消息字段设计（最小载荷）**：

| 字段 | 类型 | 说明 |
|------|------|------|
| cid | string | 会话ID |
| sender | string | 发送者ID |
| ts | int64 | 时间戳（秒级） |
| nonce | string | NaCl Box随机数（Base64） |
| cipher | string | 密文（Base64） |
| nickname | string | 发送者昵称（可选） |

设计原则：最小化暴露信息，cid、sender、nickname为明文（用于路由和展示），业务内容加密。

**关键代码**：

```go
func encryptBox(plain string, peerPub []byte, selfPriv []byte) (cipher, nonce []byte, err error)
func decryptBox(cipher, nonce []byte, peerPub []byte, selfPriv []byte) (plain []byte, err error)
```

### 核心论点（反复强调）

1. **服务端无状态无业务逻辑** - 只中转消息，不存储数据
2. **数据完全本地** - SQLite本地存储，数据主权在用户
3. **Hub去中心化** - Routes多节点集群，无单点故障
4. **不是P2P** - C/S架构但实现了去中心化优势

### 与P2P方案的区别

| 特性 | P2P方案 | 本项目 |
|------|---------|--------|
| NAT穿透 | 需要 | 不需要 |
| 连接复杂度 | 高 | 低（标准C/S） |
| 离线消息 | 难实现 | JetStream支持 |
| 数据存储 | 本地 | 本地（共同点） |
