# 摘 要

随着互联网技术的快速发展，即时通信已成为人们日常生活和工作中不可或缺的信息交互方式。然而，当前主流的即时通信应用普遍采用中心化架构，用户数据集中存储于平台服务器，存在数据主权旁落、隐私泄露风险以及单点故障等问题。尽管纯P2P方案能够实现数据本地化存储，但其面临NAT穿透困难、离线消息难以投递等技术瓶颈，限制了实际应用效果。

针对上述问题，本文设计并实现了一套基于NATS消息系统的去中心化即时通信系统。该系统采用"C/S连接、本地存储"的混合架构设计：服务端仅承担消息转发职能，不存储任何用户数据；所有聊天记录、联系人信息等敏感数据均保存在用户本地SQLite数据库中；通过NATS Routes技术构建多节点Hub集群，实现服务端层面的去中心化部署；采用Ed25519/X25519双密钥体系，结合NaCl Box端到端加密算法，确保通信内容的机密性与完整性。

本文从需求分析、总体架构、网络通信、加密安全、本地存储以及应用实现等方面对系统进行了全面阐述。系统实现了用户身份管理、端到端加密通信、离线消息同步、本地历史记录查询等核心功能。测试结果表明，该系统在保障通信安全与数据主权的前提下，具备良好的功能完整性与运行稳定性，为用户提供了一种安全、私密、自主可控的即时通信解决方案。

**关键词：** 去中心化；即时通信；NATS；端到端加密；数据主权

# Abstract

With the rapid development of Internet technology, instant messaging has become an indispensable means of information exchange in people's daily life and work. However, current mainstream instant messaging applications generally adopt centralized architecture, where user data is centrally stored on platform servers, leading to issues such as loss of data sovereignty, privacy leakage risks, and single points of failure. Although pure P2P solutions can achieve local data storage, they face technical bottlenecks such as NAT traversal difficulties and challenges in delivering offline messages, limiting their practical application.
To address these problems, this thesis designs and implements a decentralized instant messaging system based on the NATS messaging system. The system adopts a hybrid architecture of "C/S connection, local storage": the server only undertakes message forwarding functions without storing any user data; all chat records, contact information, and other sensitive data are saved in the user's local SQLite database; a multi-node Hub cluster is constructed through NATS Routes technology to achieve decentralized deployment at the server level; the Ed25519/X25519 dual-key system combined with the NaCl Box end-to-end encryption algorithm ensures the confidentiality and integrity of communication content.
This paper comprehensively elaborates on the system from aspects of requirements analysis, overall architecture, network communication, encryption security, local storage, and application implementation. The system implements core functions such as user identity management, end-to-end encrypted communication, offline message synchronization, and local history record query. Test results demonstrate that the system ensures communication security and data sovereignty while possessing good functional integrity and operational stability, providing users with a secure, private, and autonomously controllable instant messaging solution.

**Keywords:** Decentralized; Instant Messaging; NATS; End-to-End Encryption; Data Sovereignty

# 第1章 绪论

## 1.1 研究背景与意义

微信、WhatsApp等主流即时通信应用均采用中心化架构。该架构存在以下根本性问题：其一，用户数据集中存储于平台服务器，用户丧失数据主权；其二，平台可获取通信双方身份、通信时间等元数据信息；其三，服务器单点故障将导致全网服务中断。

现有去中心化方案各有不足：**纯P2P方案**（如Tox）面临NAT穿越和离线消息难题；**联邦式方案**（如Matrix）数据仍存储在服务器端。

本系统基于NATS实现去中心化即时通信：（1）服务端无状态，仅转发消息；（2）数据本地化存储；（3）Hub集群去中心化部署；（4）端到端加密。如图1所示，本系统数据完全保留在用户本地。

## 1.2 国内外研究现状

### 1.2.1 中心化方案

微信、WhatsApp等应用将用户数据存储于其运营的服务器集群。部分应用虽宣称采用端到端加密技术，但用户数据仍由平台托管，平台可获取通信双方的标识信息及通信时间等元数据。

### 1.2.2 联邦式方案

XMPP、Matrix等联邦式架构由多个独立服务器节点组成互联网络，用户可选择信任的节点进行注册。相较于中心化架构，联邦式架构赋予用户选择服务商的权利，具备一定的抗审查能力。然而，用户数据仍存储于所选服务器，数据主权问题未得到根本解决。

### 1.2.3 纯P2P方案

纯P2P方案（如Tox、Session）彻底去除服务器，用户设备直接连接。优势是数据完全本地存储，抗审查能力强。但面临**NAT穿越困难**和**离线消息难以实现**的技术挑战。

# 第2章 相关技术综述

## 2.1 即时通信架构模式

即时通信系统的架构模式可分为三类：中心化客户端/服务器（C/S）架构、联邦式架构以及纯点对点（P2P）架构。三类架构在数据存储位置、网络连接复杂度及去中心化程度上存在显著差异。

### 2.1.1 中心化C/S架构

中心化C/S架构中，客户端负责界面和本地逻辑，服务端承担业务逻辑、数据持久化和消息转发。

工作流程：用户登录时服务端验证身份并创建会话；发送消息时服务端查找接收方连接并转发；离线消息由服务端暂存。

优势：技术成熟、连接简单、功能丰富、安全可控。但根本问题是数据主权——用户数据集中存储在服务器，必须完全信赖服务提供商。

### 2.1.2 联邦式架构

联邦式架构由多个独立服务器（节点）组成网络，用户可选择信任的节点注册，节点间通过标准协议互联。

工作流程：用户注册后获得节点域内唯一标识（如user@node.com）；同节点通信流程类似中心化；跨节点通信需经双方节点转发。

XMPP和Matrix是代表性协议。联邦式架构的进步在于用户可选择服务商，网络具备抗审查性和容错性。但未解决根本问题——**用户数据仍存储在服务器上**。

### 2.1.3 纯P2P架构

纯P2P架构完全去除服务器，用户设备直接建立连接。用户上线时通过DHT广播网络地址，发送消息时通过DHT查找接收方地址并建立连接。

P2P架构的优势是数据主权（所有数据存储在本地）和抗审查能力（无法通过封锁服务器阻止通信）。

然而，纯P2P架构面临以下技术挑战：（1）NAT穿透困难：由于大多数终端设备位于私有网络内，缺乏公网IP地址，需依赖复杂的NAT穿透技术建立连接；（2）离线消息投递困难：当接收方不在线时，消息缺乏可靠的暂存机制，导致消息丢失；（3）网络拓扑复杂：消息传输路径动态变化，难以保证传输延迟的确定性和网络连接的稳定性。

### 2.1.4 本项目架构设计

本项目采用客户端-服务器连接与本地数据存储相结合的混合架构设计。服务端仅承担消息转发职能，不存储任何用户数据或业务状态。

如图2所示，本系统的架构可以概括为"C/S连接，P2P数据"——连接层面采用客户端-服务器模式，简化了网络连接；数据层面采用完全本地存储，实现了数据主权。

![图2 四种架构模式对比](images/fig02_four_arch.png)

图2 四种架构模式对比

具体来说，本系统的架构特点包括：

（1）**服务端无状态**：服务端仅作为消息传输层，不存储任何用户会话状态，不处理任何业务逻辑。每个消息都是独立的，服务端只负责根据主题将消息转发给订阅者。

（2）**数据完全本地**：所有聊天记录、联系人信息、群组数据均存储在用户本地的SQLite数据库中。服务端只中转加密后的消息，不存储消息内容。

（3）消息端到端加密，只有收发双方能解密。服务端被攻破也拿不到明文。

（4）**服务端去中心化**：通过NATS Routes技术构建多节点Hub集群，集群内节点地位平等，实现了服务端的去中心化部署。

本系统的核心设计理念是将网络连接与数据存储分离处理。连接层面采用C/S架构以确保网络稳定性；数据层面采用本地存储以保障用户数据主权。该设计兼具P2P架构的数据自主优势与C/S架构的网络连接稳定性。

## 2.2 NATS消息系统

NATS是开源高性能消息系统，采用发布/订阅模式，广泛应用于微服务通信、物联网等场景。本节介绍NATS核心技术及其在本系统中的应用。

### 2.2.1 JetStream消息系统

JetStream是NATS的消息持久化扩展模块，为NATS提供了消息存储与投递保证能力。本系统基于JetStream实现消息的可靠传输，确保消息在传输过程中不丢失，并支持离线消息的异步投递。

**JetStream核心概念**：

（1）**Stream（流）**：消息持久化的容器。本系统配置两个Stream：DChatDirect存储私聊消息（主题`dchat.dm.*.msg`），DChatGroups存储群聊消息（主题`dchat.grp.*.msg`）。

（2）Consumer从Stream中消费消息。本系统采用Pull模式，由客户端主动拉取消息，便于控制消费速率。

（3）**消息确认（ACK）**：消费者处理完消息后发送ACK确认，服务器才将消息标记为已消费。未确认的消息会被重新投递。

**本系统的消息传输流程**：

（1）应用调js.Publish发消息，存到Stream，同时推给在线订阅者；
（2）客户端订阅主题，实时收消息。
（3）离线消息：用户上线后用Pull Consumer拉取离线期间的消息，处理完发ACK确认。

JetStream的持久化能力确保了即使接收方不在线，消息也不会丢失，实现了本系统对离线消息的需求。

### 2.2.2 NATS Routes与集群架构

NATS Routes机制支持多个NATS服务器节点构建集群拓扑。在集群配置中，各节点通过Routes协议建立全连接（full-mesh）网络，消息发布到任意节点后，将自动路由并传播至集群内所有其他节点。

集群采用全mesh（full-mesh）拓扑结构，每个节点与所有其他节点建立直接连接。消息发布到任一节点后，首先通过Routes协议广播至集群内所有节点，再由各节点推送给本地订阅者。该架构具有消息传播延迟低的优势，但随着节点数量增加，节点间连接数呈O(n²)级增长（n个节点需要n(n-1)/2条连接），对网络资源消耗较大。

Routes集群的配置非常简单，只需要在配置文件中指定其他节点的地址即可：

```
cluster {
  name: "hub"
  listen: "0.0.0.0:6222"
  routes: [
    "nats-route://node1.example.com:6222"
    "nats-route://node2.example.com:6222"
  ]
}
```

Routes集群具有以下特点：

（1）**自动发现**：新节点启动后会自动与配置中的节点建立连接，并从这些节点获取集群内其他节点的信息，自动建立完整的全mesh连接。

（2）**消息广播**：发布到集群中任意节点的消息，会自动传播到集群内的所有其他节点。

（3）集群中所有节点地位对等，不存在主从关系。任一节点故障仅影响直接连接至该节点的客户端，集群整体服务保持可用，具备较高的容错能力。

（4）**水平扩展**：可以通过增加节点来扩展集群的处理能力，但需要预先配置好所有节点的连接信息。

如图3所示，本系统的Hub集群采用Routes全mesh架构，多个Hub节点地位平等，形成了去中心化的服务端架构。

### 2.2.3 LeafNode与层级扩展

LeafNode是NATS提供的层级扩展机制。在本地部署NATS Server作为LeafNode，通过LeafNode协议与远程Hub建立连接。对于本地客户端而言，连接LeafNode与直接连接Hub在功能上完全等效。

LeafNode的工作原理如下：本地NATS服务器（LeafNode）启动后，通过LeafNode协议连接到远程Hub；本地客户端连接到LeafNode，像连接普通NATS服务器一样发布和订阅消息；LeafNode自动将本地消息转发到Hub，将Hub的消息转发到本地客户端。对于本地客户端来说，连接到LeafNode与直接连接到Hub的体验完全一致。

LeafNode具有以下技术特性：（1）对本地客户端透明，可使用标准NATS客户端库进行访问；（2）支持部署于本地或内网环境，通过单一出站连接与Hub通信，无需公网IP或端口映射；（3）支持配置本地账户权限策略，实现细粒度的访问控制；（4）具备断线自动重连能力，保障连接可靠性。

在本系统中，每个用户的应用内置了一个LeafNode，本地监听127.0.0.1:4222。应用内的NATS客户端连接到本地LeafNode，LeafNode再连接到公网Hub集群。这种设计的好处是：应用内的客户端与LeafNode之间的通信完全在本地，无需认证（因为都在应用内部）；LeafNode到Hub的连接可以根据需要配置认证，保证安全性。

## 2.3 端到端加密技术

端到端加密（End-to-End Encryption，E2EE）是指消息在发送方设备上加密，在接收方设备上解密，中间传输过程中始终保持密文状态的加密方式。即使消息经过的服务器被攻破，攻击者也只能获取密文，无法获取明文内容。本节将介绍本系统采用的端到端加密技术。

### 2.3.1 NaCl Box加密方案

NaCl是Daniel J. Bernstein设计的密码学库，目标是提供简单安全的加密接口，减少开发者误用。

NaCl Box是NaCl提供的一种高级加密方案，结合了X25519密钥交换和XSalsa20-Poly1305对称加密。Box加密方案的工作流程如下：

（1）**密钥交换**：发送方生成临时的X25519密钥对，使用自己的私钥和接收方的公钥通过ECDH（椭圆曲线Diffie-Hellman）算法计算出共享密钥。

（2）利用共享密钥与随机生成的Nonce，采用XSalsa20流密码算法对明文进行加密，并通过Poly1305消息认证码算法生成认证标签，实现认证的加密（Authenticated Encryption）。

（3）**消息组装**：将临时公钥、Nonce、密文和认证标签组装成最终的消息格式发送给接收方。

（4）**解密验证**：接收方收到消息后，使用自己的私钥和发送方的临时公钥计算出相同的共享密钥，使用共享密钥和Nonce解密并验证MAC，获取明文。

安全特性：（1）前向安全性：由于每次加密均使用临时生成的X25519密钥对，即使长期私钥泄露，也无法解密历史消息；（2）认证加密机制：XSalsa20-Poly1305算法组合同时提供机密性保护（防止窃听）和完整性验证（防止篡改）。

（3）**常量时间算法**：所有加密操作都使用常量时间算法，避免时序侧信道攻击。

### 2.3.2 Ed25519与X25519密钥体系

本系统采用双密钥体系：Ed25519用于数字签名与身份认证，X25519用于密钥交换与消息加密。签名密钥与加密密钥分离使用，遵循密码学最佳实践，降低密钥泄露风险。

Ed25519是一种基于Curve25519曲线的数字签名算法，具有以下特点：

（1）**安全性高**：Ed25519被认为是对椭圆曲线离散对数问题（ECDLP）具有128位安全强度的算法，目前没有已知的有效攻击方法。

（2）Ed25519具有高效的签名与验证性能，可在通用计算平台上实现微秒级的签名验证；
（3）采用确定性Nonce生成机制，不依赖外部随机数源，消除了因随机数生成器缺陷导致私钥泄露的风险。

（4）公钥与签名长度较短（公钥32字节，签名64字节），降低了网络传输开销。

X25519是基于Curve25519曲线的密钥交换算法，用于ECDH密钥协商。Curve25519与Ed25519基于相同的底层曲线，但参数设置不同，分别优化用于密钥交换和数字签名。

本系统用Ed25519做NATS的JWT认证。用户创建身份时生成密钥对，NATS签发JWT，Hub验JWT确认身份。这种身份认证机制保证了只有合法用户才能连接到Hub。

### 2.3.3 密钥派生与安全性

本项目采用Ed25519密钥进行数字签名，但不直接用于加密运算。设计理由如下：首先，Ed25519曲线针对数字签名优化，不具备原生加密能力；其次，根据密钥分离原则，签名密钥与加密密钥应独立管理，避免单点故障导致身份认证与通信保密同时失效；最后，利用Ed25519与X25519曲线的同源特性，可从Ed25519私钥确定性派生X25519密钥，在保持身份标识一致性的同时获得ECDH密钥交换能力。

密钥派生的过程基于Ed25519和X25519曲线的同源关系。由于两种算法基于相同的底层数学结构，可以从Ed25519私钥确定性地派生出对应的X25519私钥。这种派生是单向的，无法从X25519私钥反推出Ed25519私钥。这样设计的优势是：用户只需要保存一个种子（seed），既可以用于身份认证，又可以用于消息加密；同时两种用途的密钥是分离的，即使加密密钥泄露，也不会影响身份认证的安全性。

用户私钥（包括Ed25519种子及派生的X25519私钥）仅存储于本地~/.dchat/目录，服务端不保存任何用户私钥信息，亦无法获取用户公钥（除非用户主动公开）。即使服务端遭受攻击，攻击者也无法获取用户私钥。

## 2.4 跨平台桌面开发

### 2.4.1 Wails框架介绍

Wails是构建跨平台桌面应用的开源框架，Go编写后端，Web技术构建前端。

核心架构：（1）**Go后端**：编译为原生机器码，执行效率高；（2）**Web前端**：使用React等框架，运行在嵌入WebView中；（3）**绑定机制**：Go方法与前端JavaScript自动绑定；（4）**事件系统**：支持前后端事件通信。

### 2.4.2 Wails与Electron对比

Electron框架尽管应用广泛，但存在以下技术局限：打包体积庞大（需携带Chromium内核与Node.js运行时，通常超过100MB）、内存占用较高、应用启动速度较慢。

Wails框架具有以下技术优势：体积小巧（利用系统原生WebView渲染界面）、资源占用低、启动速度快、后端性能优异（基于Go语言实现）。本系统选用Wails框架以提供更流畅的用户体验。

# 第3章 系统需求分析与总体设计

## 3.1 功能需求

本系统旨在设计并实现一个去中心化的即时通信系统，满足用户对隐私保护和数据主权的需求。通过详细的需求分析，确定系统需要实现以下核心功能：

### 3.1.1 用户身份管理

用户身份管理是系统的基础功能，包括用户注册、身份创建、密钥管理等。具体需求如下：

（1）首次使用创建身份，系统生成Ed25519签名密钥对，再派生X25519加密密钥对。

（2）私钥（种子文件）只存在本地~/.dchat/目录，文件权限限制仅当前用户可访问。

（3）**公钥分享**：用户可以导出自己的公钥信息，通过安全渠道分享给其他用户，作为建立加密通信的基础。

（4）**身份导入/导出**：用户可以将身份数据导出为备份文件，在新设备上导入恢复身份和聊天记录。

### 3.1.2 即时通信功能

即时通信是系统的核心功能，支持私聊和群聊两种模式：

（1）**私聊功能**：用户可以通过对方的公钥或用户ID发起私聊。私聊消息采用端到端加密，只有通信双方能够解密。

（2）**群聊功能**：用户可以创建群组，邀请其他用户加入。群聊消息采用对称密钥加密，群组成员共享群密钥。

（3）**消息类型**：支持文本消息，未来可扩展支持图片、文件等多媒体消息。

### 3.1.3 离线消息功能

离线消息功能保证用户在不在线时不丢失消息：

（1）**消息暂存**：当接收方不在线时，消息被暂存在Hub的JetStream中，等待接收方上线后拉取。

（2）**自动同步**：用户上线后，系统自动拉取离线期间的消息，按时间顺序展示在会话中。

（3）**消息去重**：通过消息ID进行去重，避免重复消息。

### 3.1.4 本地历史记录

本地历史记录功能提供消息的持久化存储和查询：

（1）所有收发消息自动持久化存储于本地SQLite数据库。

（2）**历史查询**：用户可以按会话查看历史消息。

（3）**消息搜索**：支持按关键词搜索消息内容，快速定位历史消息。

## 3.2 非功能需求

除了功能需求，系统还需要满足以下非功能需求，保证系统的可用性、安全性和可扩展性。

### 3.2.1 安全性需求

安全性是本系统的核心关注点，需要满足以下需求：

（1）**端到端加密**：所有私聊消息必须采用端到端加密，只有通信双方能够解密。服务端即使被攻破，也无法获取明文内容。

（2）**密钥本地存储**：用户的私钥必须仅存储在本地设备上，不得传输到服务端或任何第三方。

（3）**认证安全**：身份认证采用Ed25519数字签名，防止身份伪造。

### 3.2.2 数据主权需求

数据主权是本系统的核心设计理念，需要满足以下需求：

（1）**数据本地化**：所有用户数据（聊天记录、会话及其密钥）都存储在用户本地，无需服务端进行持久化。

（2）**用户控制**：用户对自己的数据拥有完全控制权，可以导出或删除数据。

（3）**服务端无状态**：服务端不持久化存储任何用户会话状态或业务数据，仅作为公网的消息中转节点。

### 3.2.3 可用性需求

系统需要具备良好的可用性，满足以下需求：

（1）**跨平台支持**：系统需要支持主流桌面操作系统，包括Windows，Linux和MacOS。

（2）**离线可用**：用户在无网络连接时可以查看本地历史记录，网络恢复后自动同步新消息。

（3）**故障切换**：当某个Hub节点故障时，系统可以自动切换到其他可用节点。

## 4.1 网络通信模块

### 4.1.1 应用内置的NATS LeafNode

本系统在应用中嵌入NATS Server作为LeafNode运行。该设计方案具有以下优势：（1）本地客户端与LeafNode之间的通信完全在本地进行，无需经过网络层，且无需进行身份认证；（2）LeafNode作为统一出口，集中管理与Hub集群的连接；（3）应用可完全控制LeafNode的生命周期与配置参数。

LeafNode的启动流程包括：解析配置的Hub URLs、构建NATS Server配置对象、启动内置NATS Server实例以及保存实例引用供后续管理。

### 4.1.2 NATS客户端与本地LeafNode连接

应用内的NATS客户端连接到本地LeafNode（127.0.0.1:4222）。由于是本地回环连接，无需进行身份认证，简化了连接流程。

NATS客户端的核心功能包括：建立与LeafNode的连接（支持断线自动重连）、订阅感兴趣的主题（注册消息处理回调函数）以及发布消息（将加密后的消息发送至指定主题）。

消息收发流程如下：发送消息时，将EncWire结构序列化为JSON并发布至对应主题（如dchat.dm.<cid>.msg），LeafNode接收后转发至Hub，Hub根据主题路由至目标订阅者；接收消息时，Hub将消息推送至LeafNode，LeafNode再推送至客户端，客户端调用注册的回调函数处理消息。

### 4.1.3 Hub认证机制

Hub支持两种连接认证模式：匿名连接模式与JWT凭证认证模式。

**匿名连接模式**：Hub不对连接方进行身份验证，任何LeafNode均可建立连接。该模式适用于测试环境和公开服务场景。由于应用层已启用端到端加密，即使传输层不验证身份，通信安全性仍能得到保障。

**JWT凭证认证模式**：私有部署场景下，可使用NSC（NATS Security Center）工具建立三级密钥体系（Operator、Account、User），为用户生成user.creds凭证文件。Hub配置simple_resolver.conf以启用JWT验证，LeafNode配置凭证文件路径，建立连接时自动携带JWT凭证进行身份验证。

Operator私钥（operator.nk）具有最高权限，应离线安全存储，仅用于签发Account凭证。部署时，仅需将resolver配置文件和accounts目录部署至服务器，Operator私钥保留在本地以确保安全。

### 4.1.4 Hub集群Routes配置

多个Hub节点通过Routes协议构建全mesh互联拓扑，各节点地位对等，不存在主从节点区分。

**全mesh架构**：每个Hub节点与集群内所有其他节点建立直接连接。消息发布到任一Hub后，将广播至集群内所有节点，再由各节点推送给本地订阅的客户端。该架构具有消息传播延迟低的优势，但随着节点数量增加，连接数呈指数级增长（n个节点需要n(n-1)/2条连接）。

**集群边界与限制**：当前实现采用固定集群配置。这源于技术取舍：若不支持离线消息（不用JetStream），纯NATS的Routes集群支持动态新增节点，新节点可链式连接到一个节点后自动发现集群内其他节点；若支持离线消息（使用JetStream），集群节点必须固定，新节点无法动态加入，这是由JetStream使用的Raft共识算法限制的。此外，无论哪种方式，本系统都无法突破NAT穿透实现真正的P2P通信。

**去中心化特性**：尽管集群节点数量需预先固定配置，但Hub节点间地位完全对等，不存在单点主节点。任一节点故障仅影响直连的客户端，不影响集群整体服务。可通过预先配置增加节点实现水平扩展，不同集群之间还可通过Gateway协议互联。

### 4.1.5 消息主题设计

NATS采用主题（Subject）机制进行消息路由，本系统采用分层主题命名策略。

**私聊主题**：格式为`dchat.dm.<cid>.msg`，其中cid由通信双方用户ID按字典序排序后进行哈希运算生成。该设计确保同一对用户的会话主题唯一且确定性，第三方无法预测。

**群聊主题**：格式为`dchat.grp.<gid>.msg`，其中gid为随机生成的群组标识符。群组成员订阅该主题即可接收群内消息。

**系统主题**：预留`dchat.sys.*`主题用于系统消息，如好友请求、群组邀请等。

cid生成算法：将通信双方用户ID按字典序排序后，以冒号（:）连接，计算SHA-256哈希值，取结果的前16字节转换为十六进制字符串。该算法确保无论由哪一方发起会话，生成的cid均保持一致。

## 4.2 加密与安全模块

如图11所示，系统安全架构分为四层：应用层、安全层、传输层和服务端，各层协同保障通信安全。

![图11 安全架构分层](images/fig11_security_arch.png)

图11 安全架构分层

加密与安全模块是保障系统通信安全的核心，负责密钥管理、消息加密解密等。本节将详细介绍密钥体系设计、私聊加密实现和群聊加密实现。

### 4.2.1 密钥体系

本系统采用双密钥体系架构：Ed25519用于数字签名，X25519用于消息加密。签名密钥与加密密钥分离使用，提升系统整体安全性。

**Ed25519**：基于Curve25519曲线的数字签名算法。本系统采用Ed25519进行JWT身份认证，用户创建身份时生成Ed25519密钥对，由NATS系统签发JWT凭证，Hub通过验证JWT确认用户身份。

**X25519密钥**：通过确定性算法从Ed25519私钥派生，用于ECDH密钥交换。该派生过程为单向函数，无法从X25519私钥反推出Ed25519私钥。

**密钥派生原理**：本系统采用Ed25519作为身份签名密钥（用于NATS JWT认证），但不直接使用其进行消息加密。原因如下：首先，Ed25519是专为数字签名设计的曲线，不具备直接加密能力；其次，根据密码学最佳实践，签名密钥与加密密钥应分离，避免单点失效导致既丢失身份认证又泄露通信内容；最后，通过Ed25519私钥派生X25519密钥对（基于曲线同源特性），既保持身份一致性，又获得ECDH密钥交换能力，实现端到端加密。这种设计确保即使签名密钥意外泄露，历史会话的加密消息仍无法被解密。

**曲线同源性与密钥派生技术细节**：Ed25519与X25519均基于Curve25519椭圆曲线，但采用不同的曲线参数和坐标表示。Ed25519使用扭曲Edwards形式（Twisted Edwards form）优化签名运算，X25519使用Montgomery形式优化密钥交换。两种形式可通过可逆的双有理等价变换相互转换。

密钥派生的数学过程如下：设Ed25519私钥为32字节种子，通过SHA-512哈希和剪枝运算生成签名私钥和公钥。派生X25519私钥时，提取Ed25519私钥种子的前32字节，按照X25519规范进行剪枝处理（清除低位3比特、设置第2比特为1、设置最高比特为0），得到X25519私钥。对应的公钥通过Montgomery ladder标量乘法计算。该派生过程是确定性的，同一Ed25519私钥总是派生出相同的X25519密钥对，且派生过程计算上不可逆，无法从X25519私钥反推Ed25519私钥。

**安全性分析**：双密钥分离设计提供多层安全保障。第一，功能隔离确保签名密钥泄露不影响历史通信机密性；第二，派生单向性保证即使加密私钥泄露，攻击者也无法伪造身份；第三，两种密钥使用不同的数学结构和算法参数，降低因单点算法缺陷导致整体安全崩溃的风险；第四，密钥派生过程不引入新的随机性，完全基于原始种子的确定性计算，避免因随机数生成器弱点导致的安全问题。

如图12所示，本系统采用双密钥体系，从随机种子派生Ed25519签名密钥和X25519加密密钥，实现签名与加密分离。

![图12 密钥体系设计](images/fig12_key_system.png)

**本地安全存储**：私钥文件存储于~/.dchat/目录，包括：operator.nk（Operator私钥，应离线存储）、account.nk（Account私钥）、user.seed（用户种子，可派生所有密钥）以及user.creds（JWT凭证）。文件系统权限设置为仅当前用户可访问（600权限），防止未授权访问。

### 4.2.2 私聊加密实现

私聊采用NaCl Box算法实现端到端加密，确保仅通信双方能够解密消息内容。群聊则采用AES-256-GCM对称加密算法。

![图13 加密方案对比](images/fig13_crypto_compare.png)

**加密流程**：
（1）发送方临时生成一对X25519密钥对（临时私钥和临时公钥）；
（2）使用发送方私钥与接收方公钥通过ECDH算法计算共享密钥；
（3）利用共享密钥和随机生成的Nonce，通过XSalsa20-Poly1305算法对明文进行加密，生成密文和消息认证码；
（4）将临时公钥、Nonce、密文组装为EncWire消息结构。

**消息字段设计**：

表2展示了EncWire消息的最小载荷设计。

表2 EncWire消息字段

| 字段 | 类型 | 说明 |
|------|------|------|
| cid | string | 会话ID，用于路由到正确会话 |
| sender | string | 发送者ID，明文标识发送者 |
| ts | int64 | 时间戳（秒级），用于消息排序 |
| nonce | string | NaCl Box随机数（Base64编码） |
| cipher | string | 密文（Base64编码），实际加密内容 |
| nickname | string | 发送者昵称（可选），用于展示 |

设计原则：最小化元数据暴露，仅将cid、sender、nickname字段以明文传输（用于消息路由和发送者展示），业务敏感内容全部加密。EncWire整体以明文JSON格式传输，其中nonce字段为Base64编码的随机数，用于解密cipher字段，本身非加密数据。

**解密流程**：
（1）接收EncWire消息后，从sender字段提取发送者标识；
（2）从本地好友公钥列表中查询发送者对应的公钥；
（3）使用接收方私钥、发送方公钥以及消息中的Nonce，通过NaCl Box算法解密密文；
（4）验证MAC（消息认证码）以确保消息未被篡改。

（5）将解密后的明文保存到本地数据库。

如图14所示，私聊加密流程清晰地展示了从密钥交换到消息加解密的完整过程。

### 4.2.3 群聊加密实现

群聊采用共享对称密钥机制，群组成员使用同一密钥进行加解密。

**群组密钥**：创建群组时，使用密码学安全随机数生成器生成32字节长度的AES-256-GCM对称密钥。

**密钥分发**：群组密钥通过端到端加密的私聊通道分发给群组成员。新成员加入时，由群组管理员通过私聊方式将密钥安全传输给新成员。

**群组消息加密**：使用群组密钥和随机生成的Nonce，通过AES-256-GCM算法加密消息内容。EncWire消息结构与私聊格式保持一致。

**群组消息解密**：使用群组密钥和消息中的Nonce，通过AES-256-GCM算法解密密文。所有持有群组密钥的成员均可解密。

**安全提示**：群组密钥的安全管理至关重要，密钥泄露将导致群组内所有历史消息面临解密风险。建议成员退出时轮换群组密钥以维护前向安全性（该功能在当前版本中尚未实现）。

## 4.3 本地数据存储模块

本地数据存储模块负责所有用户数据的持久化存储，是系统实现数据主权的核心。本节将详细介绍数据库设计、消息搜索与历史查询、离线消息同步的实现。

### 4.3.1 数据库设计

系统采用SQLite作为本地数据库，SQLite是嵌入式的关系型数据库，无需独立服务器进程，数据存储在单一文件中，非常适合桌面应用。

**数据库表结构**：

系统数据库包含四张核心数据表：conversations（会话表）、messages（消息表）、friend_pub_keys（好友公钥表）以及group_sym_keys（群组密钥表）。

- **conversations**：存储会话元数据，以cid为主键，包含会话类型、最后消息时间等字段；
- **messages**：存储消息内容，包含消息ID、会话ID、发送者标识、消息内容、时间戳等字段，消息内容以明文形式存储于本地；
- **friend_pub_keys**：存储好友公钥信息，包含用户ID、公钥值、创建时间等字段；
- **group_sym_keys**：存储群组对称密钥，包含群组ID、密钥值、创建时间等字段。

如图15所示，数据库E-R图展示了各表之间的关系。

![图15 数据库E-R图](images/fig15_db_er.png)

图15 数据库E-R图

**与云端方案的区别**：与传统方案将数据存储于云端服务器不同，本系统将数据存储于用户本地设备，服务端仅转发加密消息而不存储任何数据。

### 4.3.2 消息搜索与历史查询

**历史消息查询**：按会话维度分页查询历史消息，根据会话ID和偏移量参数获取指定批次的消息记录，按时间戳升序排列。支持滚动加载机制，当用户滚动至消息列表顶部时自动加载更早的历史消息。

**消息搜索功能**：系统支持基于关键词的全文检索功能。搜索时采用SQL LIKE操作符进行模糊匹配，查询内容字段中包含目标关键词的消息记录。搜索结果按时间戳排序展示，同时显示消息所属会话及发送者信息，便于用户定位消息上下文。

SQLite内置的LIKE搜索功能对于桌面应用场景已能满足基本需求。如需更强大的全文检索能力，可集成SQLite FTS（Full-Text Search）扩展模块。

### 4.3.3 离线消息同步

离线消息同步是系统的重要功能，保证用户不在线时也能接收消息。

**离线消息存储机制**：当接收方处于离线状态时，发送方发布的消息将由Hub节点的JetStream组件持久化存储。系统为每个用户创建独立的Stream（消息流）和Consumer（消费者），确保不同用户的消息数据相互隔离。

**上线同步流程**：用户上线后，系统创建Pull Consumer订阅离线消息主题，采用批量拉取策略（如每次拉取100条），逐条解密后存入本地数据库，并通知前端界面更新展示，向Hub发送ACK确认，循环执行直至无新消息可拉取。

**消息去重机制**：由于网络波动可能导致消息重复投递，系统采用消息ID进行去重判断。保存消息前查询数据库，若已存在相同消息ID则跳过该消息。

**ACK确认机制**：处理完每条消息后，系统向Hub发送ACK（Acknowledgment）确认消息。Hub收到ACK后将对应消息标记为已消费状态，最终从Stream中移除该消息。

### 4.4.2 前端界面设计

前端技术栈采用React框架、TypeScript语言以及Tailwind CSS样式库。

**界面布局**：左侧为会话列表区域（显示会话名称、最后一条消息、未读消息数），右侧为聊天区域（展示消息记录和消息输入框）。

**技术选型说明**：React用于构建用户界面组件，TypeScript提供静态类型检查，Tailwind CSS用于界面样式定义，Wails Runtime API实现前后端通信。

**前后端通信机制**：Wails框架自动将Go后端方法绑定至JavaScript前端。前端通过调用window.go.main.App.SendDirect()方法发送消息，后端通过EventsEmit()方法推送事件（如message-received事件）至前端。

### 4.4.3 核心功能实现

**消息发送流程**：前端调用SendDirect()或SendGroup()方法，后端查询目标公钥或群组密钥，执行加密运算，构造EncWire消息结构，通过NATS客户端发布至对应主题。

**消息接收流程**：NATS客户端接收到消息后调用handleIncoming()处理函数，后端执行解密并将消息存入数据库，通过EventsEmit通知前端，前端监听message-received事件并更新消息列表。

**会话管理功能**：创建新会话时生成唯一cid标识，将会话信息保存至conversations表。发送消息时自动创建对应的会话记录。前端根据会话列表数据渲染侧边栏界面。

## 5.3 安全性验证

### 5.3.1 端到端加密验证

使用NATS CLI工具订阅原始消息主题，验证消息为密文：

```bash
# 订阅私聊主题
nats sub dchat.dm.abc123.msg

# 收到的消息示例
{
  "cid": "abc123",
  "sender": "user1",
  "ts": 1714293847,
  "nonce": "8Kj7mN3pQx9vL2sT",
  "cipher": "base64encodedencrypteddata...",
  "nickname": "Alice"
}
```

验证结果：cipher字段为Base64编码的密文，无法直接读取。尝试解密时，缺少正确的私钥和发送方公钥，解密失败。验证了端到端加密的有效性。

### 5.3.2 密钥本地存储验证

检查~/.dchat/目录权限和内容：

```bash
ls -la ~/.dchat/
# 输出显示：
# -rw------- 1 user user  64 user.seed
# -rw------- 1 user user 128 user.creds
```

验证结果：私钥文件权限为600（仅所有者可读写），符合安全要求。服务端不存储任何私钥信息。

### 5.3.3 服务端无数据验证

检查Hub服务器文件系统，确认不存储用户数据：

```bash
# 在Hub服务器上执行
find /var/lib/nats -name "*.db" -o -name "*.json" | grep -E "(user|chat|message)"
# 无输出，确认无用户数据文件
```

验证结果：Hub仅存储JetStream的离线消息（密文形式），不存储任何用户身份信息或明文消息。

### 5.3.4 安全性测试结论

通过以上验证，确认系统实现了以下安全特性：

（1）端到端加密有效，服务端无法解密消息内容。

（2）私钥仅存储在本地，不会泄露到服务端。

（3）服务端无状态设计，不存储用户业务数据。

（4）即使Hub服务器被攻破，攻击者也只能获取密文和元数据，无法获取通信内容。

# 第6章 总结与展望

## 6.1 工作总结

本文设计并实现了一套基于NATS的去中心化即时通信系统。主要完成的研究工作包括：

**架构设计方面**：本系统采用客户端-服务器连接与本地数据存储相结合的混合架构。用户通过LeafNode节点连接Hub集群，所有用户数据均存储于本地SQLite数据库，服务端不保存任何用户敏感信息。该设计既保留了C/S架构的网络连接稳定性，又避免了纯P2P架构面临的NAT穿透难题。

**通信机制方面**：本系统基于NATS消息系统实现消息传输。应用内置LeafNode连接Hub集群，多个Hub节点通过全mesh拓扑互联，利用JetStream实现离线消息的持久化存储。

**安全机制方面**：本系统采用Ed25519密钥进行身份认证，通过密钥派生生成X25519密钥用于消息加密。私聊场景采用NaCl Box算法实现端到端加密，群聊场景采用AES-256-GCM对称加密算法。所有密钥仅本地存储，不上传至服务器。

**数据存储方面**：本系统使用SQLite数据库存储聊天记录、好友公钥、群组密钥等数据。支持按会话维度查询历史消息，支持基于关键词的消息搜索功能。

**应用实现方面**：本系统使用Wails框架开发跨平台桌面应用，后端采用Go语言实现，前端采用React框架构建。已在Windows和Linux平台完成测试验证。

测试结果表明，系统各功能模块运行正常，端到端加密机制有效，服务端确实不存储任何用户数据。

## 6.2 创新点

（1）服务端仅承担消息转发职能，不处理任何业务逻辑，也不存储用户数据。即使服务器遭受攻击，攻击者仅能获取密文数据，无法解密获取明文内容。

（2）在C/S架构下实现了数据主权回归用户的设计目标。聊天记录、联系人信息等数据全部本地存储，用户对自己的数据拥有完全控制权。

（3）Hub集群采用Routes协议构建全mesh互联拓扑，不存在中心节点，任一Hub节点故障不影响集群整体服务可用性。

（4）身份认证与消息加密采用分离的密钥体系。Ed25519用于数字签名与身份认证，X25519用于密钥交换与消息加密，两个密钥对从同一随机种子派生但在密码学上相互独立。

## 6.3 存在的不足

（1）部署公网Hub节点需要专用服务器资源，对个人用户而言存在一定经济成本。

（2）群聊功能相对基础，尚缺少管理员权限体系、成员管理机制等高级功能。

（3）当前仅提供桌面端应用，尚未开发移动端App。

（4）用户账号与聊天记录绑定于单一设备，更换设备时需手动导入数据恢复。

（5）当前版本仅支持文本消息传输，尚不支持图片、文件等多媒体消息。

## 6.4 未来展望

（1）开发iOS和Android移动端客户端，扩展系统至移动平台。

（2）利用NATS Gateway协议互联不同的Hub集群，构建更大规模的分布式通信网络。

（3）增加文件传输功能，支持图片、文档等多媒体消息的传输与存储。

（4）实现多设备数据同步机制，支持用户账号在电脑与手机等设备间无缝切换。

（5）集成DID（Decentralized Identifier，去中心化身份标识）技术，实现跨平台的去中心化身份认证。

（6）完善群聊功能，增加管理员权限体系、群组公告、消息撤回等功能。

（7）建立Hub共享机制，使多个用户可共用Hub集群资源，降低个体用户的使用成本。
