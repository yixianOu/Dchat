# 1. 研究背景与目标

---

## 1.1 中心化IM的困境

- **数据主权旁落** —— 用户数据集中存储于平台服务器，用户丧失控制权
- **数据暴露** —— 服务端可获取通信双方身份、时间、对话内容等数据
- **单点故障** —— 服务端宕机导致全网服务中断

---

## 1.2 现有方案对比与项目定位

| 对比维度 | 中心化 | 联邦式 | P2P | **本项目** |
|---------|--------|--------|-----|-----------|
| 数据存储 | 云端 | 所选服务器 | 本地 | **本地** |
| 服务端状态 | 有状态 | 有状态 | 无 | **无状态** |
| NAT穿透 | 不需要 | 不需要 | 需要 | **不需要** |
| 离线消息 | 完善 | 依赖服务器 | 困难 | **JetStream** |

**三点创新：**
1. 服务端零业务逻辑 —— 仅转发消息，不存数据
2. C/S架构下实现数据主权 —— 用户完全控制本地数据
3. Hub Routes对等集群 + JetStream Raft共识 —— 无中心节点，去中心化部署

# 2. 系统总体架构

---

## 2.1 技术栈选型

| 层级 | 技术选型 | 选型理由 |
|------|----------|----------|
| 消息中间件 | **NATS** | LeafNode透明代理、JetStream离线消息、Routes去中心化集群 |
| 后端语言 | **Go** | 编译型高性能、goroutine轻量并发、nats.go官方客户端 |
| 桌面框架 | **Wails v2** | 原生WebView轻量（≈15MB）、Go后端复用核心技术栈 |
| 前端 | **React + TypeScript + Tailwind** | 组件化UI、类型安全、按需生成CSS |
| 本地存储 | **SQLite** | 嵌入式单文件、零配置零运维、完整SQL查询 |
| 签名认证 | **Ed25519** | 128位安全、紧凑签名、确定性Nonce无随机数风险 |
| 密钥交换 | **X25519** (ECDH) | 同源Curve25519曲线、单向派生分离签名与加密 |
| 私聊加密 | **NaCl Box** | 临时密钥前向安全、XSalsa20-Poly1305 AEAD、API极简 |
| 群聊加密 | **AES-256-GCM** | NIST标准、Intel AES-NI硬件加速、GHASH防篡改 |

---

## 2.2 分层架构

```
┌─────────────────────────────────┐
│  UI层 (Wails: React )      │
├─────────────────────────────────┤
│  后端逻辑层 (数据存储)   │
├─────────────────────────────────┤
│  LeafNode层(nats客户端通过leafnode向hub收发消息)│
├─────────────────────────────────┤
│  Hub集群层 (Routes广播 + JetStream Raft) │
└─────────────────────────────────┘
```

- UI层：基于React构建桌面端界面，提供群聊、私聊、会话列表、密钥查询等交互gui。
- 后端逻辑层：应用内部调用nats客户端，从内嵌的leafnode获取数据。将用户会话，通信消息，好友和群聊等数据持久化到本地SQLite的数据库表中。
- LeafNode层：leafnode可以同步hub集群的特定主题数据，从leafnode查询数据，能避免向公网hub拉取重复数据。
由leafnode去连接集群，能避免因为集群单点故障导致连接到这个节点的应用客户端不可用。
- Hub 集群层：通过 Routes 协议实现节点间对等、无主从的消息广播；支持离线数据可启用 JetStream持久化数据，基于 Raft 共识算法动态选举 Leader。

# 3. LeafNode接入与身份认证

---

## 3.1 LeafNode接入与Hub集群

**LeafNode：**
- 应用内嵌leafnode节点直连Hub集群，无需公网IP或NAT穿透
- LeafNode单向同步Hub集群特定主题的数据，能向Hub集群发布消息

**Hub集群：**
- Hub之间通过Routes协议广播消息，节点对等无主从
- JetStream基于Raft共识同步元数据，动态选举Leader
- 离线消息：JetStream持久化，上线后Pull拉取同步

---

## 3.2 JWT身份认证

**三级签名链**： Operator (自签名) → Account (Operator签发) → User (Account签发)

**连接时验证签名：**
1. 前提：每个用户的Operator和Account的私钥相同，Hub集群启动时信任对应的公钥, 即可支持安全的用户认证。
2. 验UserJWT：使用Account公钥验证UserJWT签名合法性，然后取出用户公钥。
3. 验nonce签名：Hub发随机数，LeafNode用User私钥签名，Hub用公钥验签，以防止重放攻击。

**用户密钥生成：** 应用初始化时创建用户，使用NATS NKeys库自动生成Ed25519密钥对，随机数保证唯一性。

## 3.3 Ed25519 → X25519 密钥派生

**为什么需要双密钥？**
- Ed25519用于JWT数字签名与身份认证
- X25519用于ECDH密钥交换（为NaCl Box提供共享密钥）

**为什么可以单向派生？**
- Ed25519和X25519基于同一底层曲线Curve25519，只是分别用 Twisted Edwards 和 Montgomery 两种坐标形式优化签名与密钥交换，两者可通过坐标变换实现确定性的一对一映射。

**派生过程：**
1. Ed25519私钥种子(前32字节) → SHA-512哈希 → 取前32字节
2. 剪枝(Clamping)：低3位清零(防御小阶子群攻击)、最高位清零(限范围)、第254位置1(确保汉明重量)
3. 输出X25519私钥(32字节)
4. 公钥侧通过椭圆曲线坐标变换(Edwards→Montgomery)得到X25519公钥

**关键特性：** 单向不可逆 —— X25519私钥泄露不影响Ed25519身份安全

# 4. 端到端加密

---

## 4.1 私聊：加好友与密钥交换

**建立加密会话流程：**
1. A导出X25519公钥 → 通过其他可信渠道发送给B
2. B将A的公钥存入 `friend_pub_keys` 表
3. 双方用户ID字典序排序后SHA-256取前16字符 → 生成唯一cid
4. 双方自动订阅基于双方公钥生成的私聊主题
5. 发送：对方公钥 + 己方临时X25519私钥 → ECDH共享密钥 → NaCl Box加密 → NATS发布
6. 接收：NATS接收 -> 提取临时公钥 + 己方X25519私钥 → ECDH恢复共享密钥 → NaCl Box解密 + Poly1305 MAC验证 → 明文存入SQLite

---

## 4.2 私聊加密设计特点

- **算法：** NaCl Box = X25519 ECDH + XSalsa20流密码 + Poly1305消息认证
- **前向安全性：** 每次加密会话动态生成临时 X25519 密钥对。即使长期私钥发生泄露，攻击者也无法解密历史加密消息。
- **防篡改：** Encrypt-then-MAC顺序，先加密再对密文计算Poly1305认证标签，篡改者在没有共享密钥的情况下无法为篡改后的密文生成正确的认证标签。
- **AEAD：** 同时提供机密性与完整性，解密时自动验证MAC，任何篡改导致解密失败

---

## 4.3 群聊加密设计特点

**群聊加密流程：**
1. 创建群组 → crypto/rand生成32字节AES-256-GCM密钥
2. 通过私聊或者其他可信方式将密钥分发给每位成员
3. 成员将群密钥存入 `group_sym_keys` 表，自动订阅基于共享密钥生成的群聊主题
4. 发送：群消息使用共享密钥 + 随机Nonce → AES-256-GCM加密
5. 接收：从 `group_sym_keys` 取出群密钥 + 消息Nonce → AES-256-GCM解密 → GHASH验证 → 明文存入SQLite

- **算法：** 结合CTR模式的加密与GHASH的消息认证，实现高性能的对称加解密封装。
- **防篡改：** 一体化AEAD，GHASH认证标签在解密时自动验证。篡改者在没有共享密钥的情况下无法为篡改后的密文生成正确的认证标签。
- **局限性：** 共享密钥无前向安全性，当前未实现密钥轮换

# 5. 总结与展望

---

## 5.1 总结与展望

**工作总结：**
- 基于NATS实现去中心化IM，"C/S连接 + 本地数据"混合架构
- LeafNode透明接入 + Hub Routes对等集群 + JetStream(Raft)离线消息
- 私聊NaCl Box端到端加密（前向安全）+ 群聊AES-256-GCM
- Ed25519/X25519双密钥分离，签名与加密独立

**存在的不足：**
- 当前仅桌面端，无移动端支持
- 群聊无密钥轮换，退群成员理论上仍可解密
- 仅支持文本消息，不支持多媒体

**未来方向：**
- 移动端（iOS/Android）应用开发
- 群聊密钥轮换机制
- 文件/图片传输
