# 1. 研究背景与目标

---

## 1.1 中心化IM的困境

- **数据主权旁落** —— 用户数据集中存储于平台服务器，用户丧失控制权
- **数据暴露** —— 服务端可获取通信双方身份、时间、对话内容等数据
- **单点故障** —— 服务端宕机导致全网服务中断

---

## 1.2 现有方案对比与项目定位

| 对比维度 | 中心化 | 联邦式 | P2P | **本项目** |
|---------|--------|--------|-----|-----------|
| 数据存储 | 云端 | 所选服务器 | 本地 | **本地** |
| 服务端状态 | 有状态 | 有状态 | 无 | **无状态** |
| NAT穿透 | 不需要 | 不需要 | 需要 | **不需要** |
| 离线消息 | 完善 | 依赖服务器 | 困难 | **JetStream** |

**设计理念：** C/S连接 + 本地数据 = 连接稳定 + 数据主权

**三点创新：**
1. 服务端零业务逻辑 —— 仅转发消息，不存数据
2. C/S架构下实现数据主权 —— 用户完全控制本地数据
3. Hub Routes对等集群 + JetStream Raft共识 —— 无中心节点，去中心化部署

# 2. 系统总体架构

---

## 2.1 分层架构

```
┌─────────────────────────────────┐
│  应用层 (Wails: React + Go)      │
├─────────────────────────────────┤
│  NATS客户端层 (连接127.0.0.1)   │
├─────────────────────────────────┤
│  LeafNode层 (本地NATS Server)    │
├─────────────────────────────────┤
│  Hub集群层 (Routes广播 + JetStream Raft) │
└─────────────────────────────────┘
```

- 应用内嵌LeafNode，本地监听127.0.0.1:4222，无需认证
- LeafNode作为统一出口，出站连接公网Hub
- Hub之间通过Routes协议广播消息（节点对等，无主从）
- JetStream基于Raft共识同步元数据（动态选举Leader，节点固定）

---

## 2.2 数据流向与本地存储

**消息收发链路：** 明文 → 端到端加密（私聊NaCl Box / 群聊AES-256-GCM）→ NATS客户端发布 → 公网Hub转发 → NATS监听到主题消息 -> 接收方解密 → 存入本地SQLite

**本地SQLite四表：**

| 表 | 用途 |
|----|------|
| `conversations` | 会话元数据（cid、类型、最后消息时间） |
| `messages` | 消息内容（明文存储、消息ID去重） |
| `friend_pub_keys` | 好友X25519公钥 |
| `group_sym_keys` | 群组AES-256-GCM对称密钥 |

# 3. LeafNode接入与身份认证

---

## 3.1 LeafNode接入与Hub集群

**LeafNode：**
- 应用内嵌leafnode节点直连Hub集群，无需公网IP或NAT穿透
- LeafNode单向同步Hub集群特定主题的数据，能向Hub集群发布消息

**Hub集群：**
- Hub之间通过Routes协议广播消息，节点对等无主从
- JetStream基于Raft共识同步元数据，动态选举Leader
- 离线消息：JetStream持久化，上线后Pull拉取同步

---

## 3.2 JWT身份认证

**三级签名链：**

```
Operator (自签名) → Account (Operator签发) → User (Account签发)
     ↓                      ↓                      ↓
  信任锚               Hub验Account            LeafNode携带.creds
```

**连接三次验签：**
1. **验AccountJWT** —— Operator公钥验证Account签名
2. **验UserJWT** —— Account公钥验证User签名
3. **验nonce签名** —— Hub发随机nonce，LeafNode用User私钥签名（防重放）

**用户密钥生成：** 应用初始化时创建用户，使用NATS NKeys库自动生成Ed25519密钥对

---

## 3.3 Ed25519 → X25519 密钥派生

**为什么需要双密钥？**
- Ed25519用于JWT数字签名与身份认证
- X25519用于ECDH密钥交换（为NaCl Box提供共享密钥）

**派生过程：**
1. Ed25519私钥种子(前32字节) → SHA-512哈希 → 取前32字节
2. 剪枝(Clamping)：低3位清零(防御小阶子群攻击)、最高位清零(限范围)、第254位置1(确保汉明重量)
3. 输出X25519私钥(32字节)
4. 公钥侧通过椭圆曲线坐标变换(Edwards→Montgomery)得到X25519公钥

**关键特性：** 单向不可逆 —— X25519私钥泄露不影响Ed25519身份安全

# 4. 端到端加密

---

## 4.1 私聊：加好友与密钥交换

**建立加密会话流程：**
1. A导出X25519公钥 → 通过其他可信渠道发送给B
2. B将A的公钥存入 `friend_pub_keys` 表
3. 双方用户ID字典序排序后SHA-256取前16字符 → 生成唯一cid
4. 双方自动订阅基于双方公钥生成的私聊主题
5. 发送：对方公钥 + 己方临时X25519私钥 → ECDH共享密钥 → NaCl Box加密 → NATS发布
6. 接收：NATS接收 -> 提取临时公钥 + 己方X25519私钥 → ECDH恢复共享密钥 → NaCl Box解密 + Poly1305 MAC验证 → 明文存入SQLite

---

## 4.2 私聊加密设计特点

- **算法：** NaCl Box = X25519 ECDH + XSalsa20流密码 + Poly1305消息认证
- **临时密钥：** 每次加密生成临时X25519密钥对 → 实现前向安全性（长期私钥泄露不解密历史消息）
- **防篡改：** Encrypt-then-MAC顺序，先加密再对密文计算Poly1305认证标签，篡改概率 ≤ 2⁻¹⁰⁶
- **AEAD：** 同时提供机密性与完整性，解密时自动验证MAC，任何篡改导致解密失败

---

## 4.3 群聊加密设计特点

**群聊加密流程：**
1. 创建群组 → crypto/rand生成32字节AES-256-GCM密钥
2. 通过私聊或者其他可信方式将密钥分发给每位成员
3. 成员将群密钥存入 `group_sym_keys` 表，自动订阅基于共享密钥生成的群聊主题
4. 发送：群消息使用共享密钥 + 随机Nonce → AES-256-GCM加密
5. 接收：从 `group_sym_keys` 取出群密钥 + 消息Nonce → AES-256-GCM解密 → GHASH验证 → 明文存入SQLite

- **算法：** AES-256-GCM = CTR模式加密 + GHASH认证（伽罗瓦域$GF(2^{128})$通用哈希）
- **密钥生成：** crypto/rand生成32字节随机密钥
- **防篡改：** 一体化AEAD，GHASH认证标签在解密时自动验证，篡改即时拒绝
- **局限性：** 共享密钥无前向安全性，当前未实现密钥轮换

# 5. 总结与展望

---

## 5.1 总结与展望

**工作总结：**
- 基于NATS实现去中心化IM，"C/S连接 + 本地数据"混合架构
- LeafNode透明接入 + Hub Routes对等集群 + JetStream(Raft)离线消息
- 私聊NaCl Box端到端加密（前向安全）+ 群聊AES-256-GCM
- Ed25519/X25519双密钥分离，签名与加密独立

**存在的不足：**
- 当前仅桌面端，无移动端支持
- 群聊无密钥轮换，退群成员理论上仍可解密
- 仅支持文本消息，不支持多媒体

**未来方向：**
- 移动端（iOS/Android）应用开发
- 群聊密钥轮换机制
- 文件/图片传输
